
Section 4 — Phase 2: Core Routing System & Programmatic SSG Engine

4.1 SSG Strategy & Crawl-Budget Rationale

To bypass common client-rendering failures that cause indexing gaps on
Vercel deployments, the routing core of the JHB Curtain Cleaning
platform is configured as a Static Site Generation (SSG) engine. Every
marketing route — homepage, service pages, sector pages, area pages, and
blog posts — is pre-compiled into static HTML at build time. No route
that is intended for organic search visibility may use Server-Side
Rendering (SSR) on demand, because SSR introduces render-budget
consumption and timeout risks during Googlebot's first crawl pass.

The SSG strategy directly serves the architectural priority of
crawl-budget preservation defined in Section 2.1. By emitting
fully-rendered HTML at build time, the platform guarantees that every
crawl request returns a complete document on the first byte, eliminating
the JavaScript hydration dependency that has been observed to cause
"thin content" indexing penalties on competitor sites.

4.2 Target Route Schema & Indexing Strategy

The platform defines five route templates. Each template generates a
deterministic set of static paths at build time using the
generateStaticParams function. The dynamicParams flag is set to false on
every dynamic route so that requests for un-seeded URL combinations
return a 404 — this prevents crawl-budget waste on auto-generated but
non-canonical URLs.

4.3 Programmatic Parameter Generation

In Next.js 15, dynamic route parameters (params) must be handled
asynchronously. The compiler maps static paths at build time using
generateStaticParams, and the page component and metadata generator must
await the params object before destructuring. Failure to await params
will produce a TypeScript build error in strict mode and a runtime
undefined access in non-strict mode.

Listing 4.1 — Programmatic SSG Route Handler (Area Pages)

TYPESCRIPT

  // app/areas/[region]/[suburb]/page.tsx
  import { notFound } from 'next/navigation';
  import { getIndexableLocations, getSuburbsData } from
  '@/lib/integrations';
  import type { Metadata } from 'next';
  export const dynamicParams = false; // Disallows un-seeded routes to
  protect crawl budget
  interface RouteProps {
  params: Promise<{ region: string; suburb: string }>;
  }
  export async function generateStaticParams() {
  const locations = await getIndexableLocations(); // Retrieves seeded
  affluent suburbs
  return locations.map((loc) => ({
  region: loc.regionSlug,
  suburb: loc.suburbSlug,
  }));
  }
  export async function generateMetadata({ params }: RouteProps):
  Promise<Metadata> {
  const { region, suburb } = await params; // Asynchronous parameter
  extraction
  const data = await getSuburbsData(region, suburb);
  if (!data) return { title: 'Area Not Found' };
  return {
  title: `Premium Curtain Cleaning in ${data.suburbName} | JHB
  Curatorial`,
  description: `Preserve luxury drapes with our scientific in-situ

Critical implementation notes:

-   `dynamicParams = false` is non-negotiable. It guarantees that any
    URL combination not returned by generateStaticParams produces a 404
    response rather than a runtime render. This protects the crawl
    budget from being consumed by infinite URL-space permutations.

-   `params: Promise<...>` is the Next.js 15 type signature.
    Destructuring must occur after await params. The pre-15 pattern of
    synchronous destructuring ({ region, suburb } = params) is a type
    error.

-   `generateMetadata` must be exported as an async function and return
    a Promise<Metadata>. The canonical URL must use the
    alternates.canonical field, not a top-level canonical key.

-   `getIndexableLocations()` retrieves the seeded affluent suburb list
    from @/lib/integrations. This function reads from the Prisma
    database at build time; the build environment must have DATABASE_URL
    configured with read access to the production-seeded suburbs table.

4.4 Sitemap & Canonical URL Discipline

Every route generated by generateStaticParams must also be present in
sitemap.xml. The sitemap is generated by app/sitemap.ts using the same
getIndexableLocations() data source to guarantee one-to-one parity
between indexed routes and sitemap entries. Any mismatch between
generateStaticParams output and sitemap.xml entries will trigger a
quality-gate failure (see Section 12).

Canonical URLs are emitted via the alternates.canonical field in
generateMetadata. The canonical URL must match the absolute URL of the
rendered page, including the https://www.jhbcurtaincleaning.co.za origin
prefix. Relative canonical URLs or canonicals omitting the www.
subdomain are prohibited because they cause Google to discard the
canonical signal.

Section 5 — Phase 3: Schema Design & Prisma Database Layer

5.1 Database Strategy & Persistence Model

Data operations for the JHB Curtain Cleaning platform are centralized in
a PostgreSQL 15+ deployment. Database schemas are compiled and migrated
using Prisma ORM to structure the three primary data domains: lead
capture, partner referral lifecycle, and subscriber newsletter
management. PostgreSQL is selected over MySQL or SQLite because of its
superior JSONB indexing performance (required for the analytics-driven
lifecycle segmentation), native Decimal type for commission currency
precision, and robust row-level security support for future multi-tenant
expansion.

The Prisma client is instantiated once per process and shared across
route handlers and Bun microservices. Connection pooling is handled by
Prisma's built-in PgBouncer-compatible connection manager. The
DATABASE_URL environment variable must use the postgresql:// scheme and
include the ?schema=public&pgbouncer=true&connection_limit=5 query
parameters when deployed to serverless environments to prevent
connection exhaustion during cold starts.

The schema file prisma/schema.prisma defines the data source, client
generator, enumerated types, and three primary models. The LeadStatus
enum governs the partner-referral lifecycle from initial capture through
commission settlement and archival. The Partner, Lead, and Subscriber

5.4 Migration Workflow

Schema changes are applied through the Prisma migration workflow. The
following commands form the canonical migration sequence and must be
executed in this order:

10. Author the schema change in prisma/schema.prisma.

11. Generate the migration locally: bunx prisma migrate dev --name
    <descriptive_name> --create-only. This produces a timestamped SQL
    file under prisma/migrations/ without applying it.

12. Review the generated SQL to confirm no destructive operations (e.g.,
    DROP COLUMN) are present without explicit team approval.

13. Apply the migration locally: bunx prisma migrate dev. This applies
    the SQL and regenerates the Prisma client.

14. Commit the migration files to version control. The migration history
    is the source of truth for production schema state.

15. Apply in CI/CD: bunx prisma migrate deploy. This command applies all
    pending migrations in order and is idempotent.

prisma db push is prohibited in any environment other than local
development. It bypasses migration history and produces schema drift
between the migration files and the live database, which breaks
reproducible deployments.

Section 6 — Phase 4: Port-Isolated Bun Microservices (API Layer)

6.1 Microservice Decomposition Rationale

To optimize performance and isolate computational workloads, the JHB
Curtain Cleaning platform decouples high-frequency partner-ledger
processing and email-campaign automation into independent Bun services
running on dedicated ports. The Next.js application core (port 3000)
remains responsible for page rendering, form capture, and AI route
handlers; the Bun Partner Service (port 3001) owns the commission
calculation ledger; and the Bun Email Service (port 3002) owns
newsletter dispatch and lifecycle automation.

The decomposition is driven by three engineering constraints. First, the
commission calculation is a CPU-bound operation that, if executed
in-process with the Next.js request handler, would inflate the response
latency of lead-capture form submissions beyond the 200ms Core Web
Vitals budget for INP. Second, the email campaign dispatch is an
I/O-bound long-running operation that benefits from Bun's native TLS and
DNS-resolver caching, which outperform Node's nodemailer pipeline by
approximately 3× on cold start. Third, isolating these services on
separate ports allows independent horizontal scaling: if a
partner-promotion spike increases commission-calculation load, only port
3001 needs to scale out, leaving the Next.js core and email service
untouched.

6.2 Service Topology & Caddy Routing

All three services sit behind a single Caddy reverse proxy that
terminates TLS on port 443 and routes traffic based on hostname and path
prefix. Caddy is selected over Nginx or Traefik because of its automatic
HTTPS certificate provisioning via Let's Encrypt, its declarative
Caddyfile syntax, and its native WebSocket upgrade support for the
future chatbot gateway.

return 404 and break the affiliate dashboard.

6.3 Commission Ledger Formula

The commission ledger uses a standardized formula to calculate dynamic
partner referral payouts. The formula is intentionally simple — the
product of the pre-tax invoice value and the partner's commission rate —
because complexity in commission calculation is the leading cause of
partner disputes. The rate is stored as a percentage (0–100) on the
Partner.commissionRate field; the formula divides by 100 to convert to a
multiplier before applying it to the invoice value.

Formula 6.1 — Commission Calculation

  CommissionDue = InvoiceValue × (Partner.commissionRate / 100)

Where:

-   CommissionDue is the computed commission payable to the partner,
    stored on Lead.commissionDue.

-   InvoiceValue is the pre-tax sales invoice value, stored on
    Lead.invoiceValue.

-   Partner.commissionRate is the designated percentage rate, stored on
    Partner.commissionRate (default 10.00).

The formula is executed by the Partner Engine service when a Lead
transitions from CAPTURED to COMMISSION_PENDING. The result is persisted
to Lead.commissionDue as a Decimal(10,2) to preserve currency precision.
Floating-point arithmetic is prohibited for currency operations; the
Partner Engine uses parseFloat only for the rate division and Decimal
arithmetic for the multiplication and storage.

6.4 Partner Engine Service Implementation

The Partner Engine is a Bun-native HTTP service that exposes a single
POST /api/partner/calculate endpoint. The endpoint accepts a JSON
payload containing leadId and invoiceValue, validates that the Lead is
associated with a Partner, applies Formula 6.1, and persists the result
with the Lead's status transitioned to COMMISSION_PENDING.

Listing 6.1 — Partner Engine Service
(`services/partner-engine/index.ts`)

TYPESCRIPT

  // services/partner-engine/index.ts
  import { PrismaClient } from '@prisma/client';
  const prisma = new PrismaClient();
  const PORT = 3001;
  const server = Bun.serve({
  port: PORT,
  async fetch(req) {
  const url = new URL(req.url);
  if (req.method === 'POST' && url.pathname ===
  '/api/partner/calculate') {
  const { leadId, invoiceValue } = await req.json();
  const lead = await prisma.lead.findUnique({
  where: { id: leadId },
  include: { partner: true }
  });
  if (!lead || !lead.partner) {
  return new Response(
  JSON.stringify({ error: 'Referral tracking invalid' }),
  { status: 404 }
  );
  }
  // POPIA hard gate: refuse to process leads without explicit consent
  if (!lead.popiaConsent || !lead.consentTime) {
  return new Response(
  JSON.stringify({ error: 'POPIA consent required before processing' }),
  { status: 403 }
  );
  }
  const rate = lead.partner.commissionRate / 100;
  const commission = parseFloat(invoiceValue) * rate;
  const updatedLead = await prisma.lead.update({
  where: { id: leadId },
  data: {
  invoiceValue: parseFloat(invoiceValue),
  commissionDue: commission,
  status: 'COMMISSION_PENDING'
  }
  });
  return new Response(
  JSON.stringify({ success: true, commission, updatedLead }),
  { status: 200 }
  );
  }
  return new Response('Not Found', { status: 404 });
  }
  });
  console.log(`Partner Engine Online: Port ${PORT}`);

Critical implementation notes:

-   POPIA hard gate (lines 19–24): The service refuses to process any
    Lead where popiaConsent = false or consentTime IS NULL. This gate is
    redundant with the application-layer gate defined in Section 5.3,
    but defense-in-depth is required because the Partner Service runs as
    an independent process with its own Prisma client and could be
    invoked directly by an internal cron job that bypasses the Next.js
    request pipeline.

-   `include: { partner: true }` performs a JOIN to fetch the partner
    record in a single round-trip. Without this directive, the service
    would issue two queries and introduce race-condition risk if the
    partner record is updated between the two reads.

-   `parseFloat(invoiceValue)` is used because JSON natively transmits
    numbers as doubles; the conversion to Prisma's Decimal type happens
    at the ORM layer. For invoices exceeding R100,000, the engineering
    team must verify that the float-to-decimal conversion does not
    introduce rounding error beyond the second decimal place.

-   `Bun.serve` is the Bun-native HTTP server constructor. It does not
    require Express or Hono; the routing is implemented manually with if
    guards because the service exposes only a single endpoint. If
    additional endpoints are added in future phases, migrate to Hono for
    middleware support.

Section 7 — Phase 5: Local SEO, AEO & GEO Structured Markups

7.1 Phase 5 Scope & Optimization Pillars

Phase 5 implements the three optimization pillars that govern
discoverability of the JHB Curtain Cleaning platform across traditional
search engines, AI-powered answer engines, and generative AI citation
systems. Local SEO targets Google's local pack and map results through
suburb-specific landing pages with hyperlocal landmarks and pain points.
Answer Engine Optimization (AEO) targets AI-powered answer engines
(Google AI Overviews, ChatGPT Search, Perplexity) through
question-focused headings and concise 45-word direct-answer blocks.
Generative Engine Optimization (GEO) targets large language model
citation behavior through structured JSON-LD schema that links the
service profile directly to Google's local search graph.

All three pillars share a single content substrate: the suburb, sector,
and service landing pages generated by the SSG engine in Phase 2. Phase
5 does not create new routes; it layers structured data and copywriting
templates onto the existing route inventory. This separation of concerns
is intentional — it allows the SEO team to revise content and schema
without touching routing code, and it allows the engineering team to
refactor routing without disturbing SEO assets.

quoted verbatim by ChatGPT and Perplexity without truncation.

Key AEO copywriting rules:

-   Question phrasing — The H2 heading must be phrased as a complete

    speak to a voice assistant. Avoid keyword-stuffed questions
    ("Curtain Cleaning Sandton Price?") in favor of conversational
    questions ("How much does curtain cleaning cost in Sandton?").

-   45-word answer limit — The answer paragraph must be between 40 and

    40–50 word range.

    entries in the References section (Section 16) and are stripped from
    the visible rendering but preserved in the JSON-LD mainEntity field
    for FAQPage schema.
